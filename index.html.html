📡 What is CRLF Injection?

CRLF stands for Carriage Return (\r) + Line Feed (\n) — the two-character sequence used to terminate lines in HTTP headers. When an attacker can inject %0d%0a (URL-encoded CRLF) into a value that gets reflected into an HTTP response header, they can:

  • Inject arbitrary HTTP response headers
  • Perform HTTP response splitting to poison caches
  • Set malicious Set-Cookie headers → session fixation
  • Inject XSS payloads into a fake second response body
  • Bypass security headers (CSP, X-Frame-Options, etc.)

🔬 How It Works — Step by Step

Consider a redirect endpoint that mirrors user input:

GET /redirect?url=https://example.com HTTP/1.1 Host: vulnerable-site.com → Server responds with: HTTP/1.1 302 Found Location: https://example.com

Now inject %0d%0aSet-Cookie:%20session=evil as part of the URL:

GET /redirect?url=https://example.com%0d%0aSet-Cookie:%20session=evil → Server reflects the injected bytes into the header: HTTP/1.1 302 Found Location: https://example.com Set-Cookie: session=evil ← INJECTED!

💀 Attack Vectors

  • Response Splitting: Inject a full second HTTP response to poison a shared cache
  • XSS via Headers: Inject Content-Type: text/html + blank line + <script> body
  • Cookie Injection: Set arbitrary cookies → session fixation / CSRF bypass
  • Log Injection: Inject newlines into server logs to forge or hide log entries
  • Security Header Bypass: Override CSP, X-Frame-Options, HSTS by injecting replacement headers

🛡️ Mitigations

  • Strip or reject \r and \n from any user input used in HTTP headers
  • Use framework APIs to set headers — they encode values automatically
  • Validate redirect URLs against an allowlist
  • Encode output before placing it in a header context
  • Deploy a WAF rule that blocks %0d / %0a in header-reflected parameters
